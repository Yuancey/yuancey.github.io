报错

 ERR_PNPM_ADDING_TO_ROOT  Running this command will add the dependency to the workspace root, which might not be what you want - if you really meant it, make it explicit by running this command again with the -w flag (or --workspace-root). If you don't want to see this warning anymore, you may set the ignore-workspace-root-check setting to true.

只有共享的模块才可以安装到根目录下，非共享的模块应该安装到某个模块下。

强制安装 pnpm intall vue -w 将当前依赖包安装到根目录下

默认情况下，pnpm 安装的依赖包会通过软链接的方式放在.pnpm文件夹下。（为了复用）

pnpm中：a –> b –> c

如果相单独引用b, import 'b'无法正常引用，所以需要将.pnpm文件下的依赖包拍平，重新放到node_modules下

平时引用就可以和node_modules的引用方式一样，直接引用了

实现方式：

创建.npmrc文件

//羞耻的提升
shamefully-hoist = true

重新安装依赖

pnpm install

被拍平，可以通过node_modules的方式去引用包

3.2.1.1 如何正确引用packages/shared/src/index.ts文件

packages/reactivity/src/index.ts

实现方法：通过ts解析依赖关系

安装ts

npm install typescript -D

生成配置文件

tsc --init

tsconfig.json 添加常用的配置

{
  "compilerOptions": {
    "outDir": "dist", // 输出的目录
    "sourceMap": true, // 采用sourcemap
    "target": "es2016", // 目标语法
    "module": "esnext", // 模块格式
    "moduleResolution": "node", // 模块解析方式
    "strict": false, // 严格模式
    "resolveJsonModule": true, // 解析json模块
    "esModuleInterop": true, // 允许通过es6语法引入commonjs模块
    "jsx": "preserve", // jsx 不转义
    "lib": ["esnext", "dom"], // 支持的类库 esnext及dom
  }
}

tsconfig.json修改配置–>实现指定引用文件位置

{
    "baseUrl": "./",  //  ./以当前项目为根目录，当引用@vue时，会映射到新的路径
    "paths": {
        "@/vue/*": ["packages/*/src"]
    }
}

卸载vue

pnpm uninstall vue

此时，引用正常

packages/reactivity/src/index.ts

import { isObject } from "@vue/shared";

console.log(isObject({ a: 1 }))  //需要将ts打包成ts才可以运行

3.3 生成环境rollup打包

3.3.1 安装esbuild

pnpm install esbuild -D -w

3.3.1 安装node

npm i --save-dev @types/node

3.3.2 创建文件

scripts/dev.js

//import node不支持 当前是common.js语法
// js文件 使用require不用解析
const { build } = require('esbuild')
const { resolve } = require('path')

const target = 'reactivity'

build({
  entryPoints: [resolve(__dirname, `../packages/${target}/src/index.ts`)],
  outfile: resolve(__dirname, `../packages/${target}/dist/${target}.js`),
  bundle: true, //将依赖的模块全部打包
  sourcemap: true, //支持调试
  format: 'esm', //打包出来的时esm es6模块
  platform: 'browser', //打包的结果给浏览器使用
  watch: { // 监控文件变化
    onRebuild() { //文件变化时重新构建
      console.log(`rebuild~~~~`)
    }
  }
})
.then(() => {
  console.log('watching~~~');
})

3.3.3 配置命令脚本

package.json

"scripts": {
    "dev": "node scripts/dev.js"
 },

3.3.4 运行

npm run dev

4.1 monkrepo

多项目 多包依赖 依赖包关系

Monorepo 是一种项目代码管理方式，指单个仓库中管理多个项目，有助于简化代码共享、版本控制、构建和部署等方面的复杂性。

适用性：

主要解决大仓库 Monorepo 构建效率低的问题。适用于包非常多、代码体积非常大的 Monorepo 项目。

使用问题：

  • npm/yarn 安装时，会出现【幽灵依赖】现象
  • 多项目的多包情况下 会出现冗余依赖

解决：pnpm 软链接

pnpm-workspace.yaml

对依赖的管理，将依赖提升到node_modules目录下

优点：

  • 减少冗余依赖

  • 解决幽灵依赖

  • 引用node_modules下的依赖包更方便

  • 存储中心 Store 集中管理依赖：不同项目，相同版本依赖安装只进行硬链接；不同版本依赖，只增加Diff文件

  • 项目 package.json 依赖列表，和node_modules/.pnpm目录结构一致

  • 相同依赖安装时，将 Store 中的依赖硬链接到项目的 node_modules/.pnpm 下，而不是复制，速度快

  • 项目node_modules中已有依赖重复安装时，会被软链接到指定目录下

  • 存储中心 Store 集中管理依赖：不同项目，相同版本依赖安装只进行硬链接；不同版本依赖，只增加Diff文件

4.2 pnpm

pnpm原理

1、存储中心Store集中管理依赖

不同项目，相同版本依赖安装建立硬链接，和node_

modules/.pnpm目录结构一致

2、项目package.json依赖结构和node_modules/.pnpm目录结构一致

3、相同依赖安装时，将store中的硬依赖链接到项目的node_modules/.pnpm下，而不是复制，速度快

4、项目node_modules中已有依赖重复安装时，会被软连接到指定目录下

4.3 硬链接和软链接（符号链接）的区别

1链接内容不同：硬链接仅能链接文件，而符号链接可以链接目录

2关联内容不同：硬链接在链接完成后仅和文件内容关联，和之前链接的文件没有任何关系。而符号链接始终和之前链接的文件关联，和文件内容不直接相关。
